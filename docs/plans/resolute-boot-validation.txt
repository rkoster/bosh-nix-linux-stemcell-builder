IMPORTANT (LXD/Incus CPI quirk found during upload): `bosh upload-stemcell` fails
with CPI error 'Missing creation date' in create_stemcell when the tarball is
fully deterministic (all mtimes @ epoch 0). The LXD CPI derives the image
creation_date from the ModTime of the first >100MiB tar entry — which for the
openstack stemcell is `root.img` *inside* the nested `image` tar.gz. Operator
workaround (does NOT change the committed deterministic build): repack a deploy
copy with a real, non-zero mtime on the inner `root.img`. See the deploy runbook.

--------------------------------------------------------------------------------
Prior FAIL (2026-07-17, before fix) — retained for history
--------------------------------------------------------------------------------
Earlier both disks FAILED to boot with a root-mount kernel panic:
  [    3.962220] Waiting 300 sec before mounting root device...
  [  334.081849] VFS: Cannot open root device "UUID=4444...4444" or unknown-block(0,0): error -6
  [  334.098270] Kernel panic - not syncing: VFS: Unable to mount root fs on unknown-block(0,0)
Symptom: "Freeing initrd memory: 240K" — a tiny, microcode-only initrd with no /init.

Root cause (CONFIRMED and FIXED): the initramfs repack loop in
build/stemcells/bootable-rootfs.sh extracted each initrd with `cpio -idm < img`,
which reads only the FIRST cpio segment (CPU microcode) and stops at its TRAILER,
silently discarding the remaining uncompressed early segments AND the entire
compressed main initramfs (with /init and the modules that mount root). The
resulting microcode-only initrd left the kernel's built-in mounter unable to
resolve root=UUID=... -> panic. This affected BOTH Noble and Resolute; it was
never caught because the production path was only determinism-tested, never
boot-tested.

Fix: the repack now splits each initrd at the early/main boundary (same header
walk as unmkinitramfs(8)), preserves the uncompressed early (microcode) segments
byte-for-byte, and re-packs ONLY the decompressed main initramfs deterministically
(sorted names, mtime @0, owner 0:0, newc, gzip -n), then concatenates them back.
Verified: reassembled initrd is path-list- and content-identical to the original
(only cpio framing metadata normalized), full-size, and boots to login on both
releases. All Noble baseline hashes were re-anchored against the corrected,
actually-booting image (docs/plans/baseline-hashes.txt).
